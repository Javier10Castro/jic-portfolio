# Testing Guide — Vercel API Endpoints

Base URL: `https://web-portfolio-kappa-wheat.vercel.app`

---

## Prerequisites

- **PowerShell 5.1+** (Windows) or **PowerShell 7+** (cross-platform)
- Use `Invoke-RestMethod` — do NOT use `curl.exe`. PowerShell interprets `{}` in curl arguments, causing `unmatched brace` errors.

---

## submittedAt Requirement

Every POST request **must** include a `submittedAt` field — a Unix millisecond timestamp generated by the frontend when the user submits the form.

```javascript
// Frontend — generate before sending
const submittedAt = Date.now();
```

### Rules

| Condition | Result | Error |
|---|---|---|
| Missing `submittedAt` | 400 | `INVALID_REQUEST` |
| `submittedAt > now + 10s` (future clock skew) | 400 | `INVALID_REQUEST` |
| `submittedAt < now - 2h` (stale/replay) | 400 | `INVALID_REQUEST` |
| Valid timestamp | passes | — |

### Why?

- Prevents automated replay attacks (stale check)
- Detects clock-skewed clients or bots (future check)
- Validates that the form was actually rendered to a human (timing check)

---

## Response Codes

| Code | Meaning | When |
|---|---|---|
| `202` | Accepted — queued for background SMTP delivery | Valid request, queued |
| `400` | Invalid input | Bad body, missing/`submittedAt`, validation fail |
| `405` | Method not allowed | GET/PUT/DELETE on POST-only endpoints |
| `429` | Rate limited | IP burst or email dedup window |
| `503` | Queue overflow | Too many pending jobs (limit: 100) |
| `500` | Server error | Missing SMTP credentials, unexpected failures |

## Rate Limiting Tests

### Layer 1: IP-based burst (429 for hard limit)

The hard limit is 60 requests per IP per 60 seconds. Send 12 rapid requests with the same email:

```powershell
$url = "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact"

1..12 | ForEach-Object {
  $body = @{
    name        = "Burst Test"
    email       = "burst-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
    message     = "Burst test $_"
    submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
  } | ConvertTo-Json

  try {
    $r = Invoke-RestMethod -Uri $url -Method POST -ContentType "application/json" -Body $body
    Write-Host "$_ — 202 queued"
  } catch {
    $statusCode = $_.Exception.Response.StatusCode.value__
    Write-Host "$_ — $statusCode"
  }
}
```

**Expected:** After the first few requests, some return `429 RATE_LIMITED`.

---

### Layer 2: Email dedup (429 within 5 minutes)

Submit the same email twice within 5 minutes:

```powershell
$body = @{
  name        = "Dedup Test"
  email       = "dedup-test-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
  message     = "First attempt"
  submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
} | ConvertTo-Json

# First request — should succeed
Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
  -Method POST `
  -ContentType "application/json" `
  -Body $body

# Second request with same email immediately after — should 429
try {
  Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
    -Method POST `
    -ContentType "application/json" `
    -Body $body
} catch {
  $_.Exception.Response.StatusCode.value__  # Should be 429
}
```

**Expected:** First = `202`, Second = `429`

### Reading response headers

`Invoke-RestMethod` returns the body only. To inspect response headers (like `X-Request-Id` or rate limit headers):

```powershell
$url = "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact"
$body = @{
  name        = "Headers Test"
  email       = "headers-test-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
  message     = "Check headers"
  submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
} | ConvertTo-Json

$response = Invoke-WebRequest -Uri $url -Method POST -ContentType "application/json" -Body $body

$response.Headers["X-Request-Id"]
$response.Headers["X-Queue-Id"]
$response.Headers["X-RateLimit-Remaining"]
$response.Headers["Retry-After"]
$response.Headers["X-Deploy-SHA"]
```

**Available headers:**

| Header | Present on | Purpose |
|---|---|---|
| `X-Request-Id` | All responses | Trace ID for correlating logs |
| `X-Deploy-SHA` | All responses | Current Git SHA deployed |
| `X-Deploy-Env` | All responses | `production`, `preview`, or `local` |
| `X-Body-Parse-Method` | All responses | How the body was parsed (`object`, `string`, `stream`) |
| `X-Queue-Id` | 202 only | Position in the processing queue |
| `X-Queue-Depth` | 202 only | Total items in queue |
| `X-RateLimit-Limit` | 429 only | Max requests allowed in window |
| `X-RateLimit-Remaining` | 429 only | Requests remaining in window |
| `Retry-After` | 429 only | Seconds to wait before retrying |
| `X-RateLimit-Soft` | 202 only | `1` if IP exceeds soft limit (30 req/60s) |

## Common Mistakes (Windows)

### Mistake 1: Using curl

```powershell
# ❌ DO NOT DO THIS — PowerShell interprets {} before curl sees them
curl -X POST https://... -d '{"name":"Test"}'
```

```
# Error: unmatched brace
```

**Fix:** Use `Invoke-RestMethod` with a PowerShell object and `ConvertTo-Json`.

---

### Mistake 2: Invoke-WebRequest on non-2xx

```powershell
# ❌ This throws on 400/429/503
$r = Invoke-WebRequest -Uri $url -Method POST -ContentType "application/json" -Body $body
```

```
# Error: The remote server returned an error: (429) Too Many Requests
```

**Fix for handling errors:**
- Use `Invoke-RestMethod` (throws on non-2xx, but returns body in `$_.Exception.Response`)
- Use `try/catch` with `$_.Exception.Response.StatusCode.value__`
- Use Node.js `fetch()` which doesn't throw on non-2xx

---

### Mistake 3: Using `[DateTime]::Now` instead of UTC

```powershell
# ❌ Wrong — local time varies by timezone
$submittedAt = [DateTime]::Now

# ✅ Correct — UTC millisecond timestamp
$submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
```

---

### Mistake 4: Forgetting submittedAt

```powershell
# ❌ Returns 400 INVALID_REQUEST
$body = @{ name="Test"; email="test@t.com"; message="Hi" } | ConvertTo-Json

# ✅ Include submittedAt
$body = @{ name="Test"; email="test@t.com"; message="Hi"; submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds() } | ConvertTo-Json
```

---

### Mistake 5: Reusing the same email in rapid succession

The email dedup window is **5 minutes**. If you test with the same email twice within 5 minutes, the second request gets `429 RATE_LIMITED`. Use timestamp-unique emails for repeated testing:

```powershell
$uniqueEmail = "test-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@example.com"
```

## Verifying Async Delivery (Queue)

A `202` response means the request was accepted and queued. Emails send in the background (usually within 0-3s). To verify delivery:

1. Check the inbox at `GMAIL_USER` — both admin and client emails should arrive within seconds.
2. Enable debug logging to see queue processing:
   ```powershell
   $env:DEBUG_API="true"
   ```
   This logs queue lifecycle: `queue_enter`, `queue_exit`, `smtp_attempt`, `smtp_success`, `smtp_retry`.
 3. Check Vercel Function logs via `vercel logs --follow`.

## Load Testing Strategy

### Rules of engagement

Rate limit operates BEFORE queue admission. 429 errors are NOT queued. A load test that triggers rate limiting never tests queue performance — it only tests gateway limits.

### Baseline Test — Single Request Validation

```powershell
$body = @{
  name        = "Load Baseline"
  email       = "baseline-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
  message     = "Single request validation"
  submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
} | ConvertTo-Json

Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
  -Method POST -ContentType "application/json" -Body $body
```

**Expected:** 202, position 0, depth 0.

### Controlled Throughput Test — Queue Stability (250ms delay)

```powershell
1..10 | ForEach-Object {
  $body = @{
    name        = "Controlled Load"
    email       = "load-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
    message     = "Request $_"
    submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
  } | ConvertTo-Json

  try {
    $r = Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
      -Method POST -ContentType "application/json" -Body $body
    Write-Host "$_ — 202 (position=$($r.position), depth=$($r.depth))"
  } catch {
    $code = $_.Exception.Response.StatusCode.value__
    Write-Host "$_ — $code"
  }

  Start-Sleep -Milliseconds 250
}
```

**Expected:** All 202. Position increases with queue depth, depth returns to 0 as workers drain. Validates queue stability without triggering rate limit.

### Progressive Stress Test — Rate Limit Threshold Discovery

```powershell
$found = $false
$i = 1
while (-not $found) {
  $body = @{
    name        = "Stress Test"
    email       = "stress-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
    message     = "Request $i"
    submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
  } | ConvertTo-Json

  try {
    $r = Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
      -Method POST -ContentType "application/json" -Body $body
    Write-Host "$i — 202"
    $i++
  } catch {
    $code = $_.Exception.Response.StatusCode.value__
    Write-Host "$i — $code (rate limit reached)"
    $found = $true
  }

  Start-Sleep -Milliseconds 100
}
```

**Expected:** First N requests succeed (202), then a 429 appears when the hard limit is reached. The count before 429 reveals the effective per-instance window capacity.

### Fail-Fast Test — Gateway Limit Discovery (WARNING)

```powershell
# WARNING: This WILL trigger 429 errors rapidly.
# This does NOT test queue performance — it tests gateway limits only.
1..20 | ForEach-Object {
  $body = @{
    name        = "Fail Fast"
    email       = "fast-$([DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds())@test.com"
    message     = "Request $_"
    submittedAt = [DateTimeOffset]::UtcNow.ToUnixTimeMilliseconds()
  } | ConvertTo-Json

  try {
    $r = Invoke-RestMethod -Uri "https://web-portfolio-kappa-wheat.vercel.app/api/sendContact" `
      -Method POST -ContentType "application/json" -Body $body
    Write-Host "$_ — 202"
  } catch {
    $code = $_.Exception.Response.StatusCode.value__
    Write-Host "$_ — $code"
  }
}
```

**Expected:** Saturation within the first few requests. Most return 429. Queue depth stays near 0 because rate limiting blocks admission.

### Why Uncontrolled Loops Are Harmful

| Problem | Explanation |
|---|---|
| Immediate 429 saturation | First requests consume the window, remainder are rejected — no meaningful data on queue behavior |
| Does NOT test queue performance | Queue only sees requests that pass rate limiting. A burst loop that triggers 429 tells you nothing about queue throughput |
| Misleading failure rate | A test reporting 80% failure rate due to rate limiting is not a system failure — it is expected gateway behavior |

## Client Retry & Backoff Strategy

The contact form retries 429 responses automatically with exponential backoff (0s, 1s, 2s, 4s, max 4 total).

### Testing Client Retry

1. Trigger a 429 by hitting the rate limit (saturate IP window with rapid requests).
2. Observe the browser console logs: each retry emits `{ requestId, retryAttempt, retryDelayMs }`.
3. The UI should show `"Retrying... (Attempt 2 of 4)"` while retries are in flight.
4. After all 4 retries fail (or a retry succeeds), the UI resolves to success or error.

### Important

- Client retry is a **UX feature**, not a reliability mechanism — each retry must independently pass the server's rate limit gate.
- The 7s worst-case window means a sustained 429 condition will exhaust all 4 retries and surface an error.
- Network errors (no connection, DNS failure, timeout) do not trigger retry — only HTTP 429 does.

### Summary

| Test Type | Delay | Goal | Expected Result |
|---|---|---|---|
| Baseline | none | Single request validation | 202 |
| Controlled throughput | 250ms | Queue stability | All 202, depth varies |
| Progressive stress | 100ms | Rate limit threshold | N×202, then 429 |
| Fail-fast (WARNING) | none | Gateway limit discovery | Mostly 429 |
